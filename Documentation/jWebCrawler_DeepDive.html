Class Hierarchy & Relationships

jWebCrawler has zero inheritance, zero interfaces, and zero abstract classes. Every class is a concrete, standalone POJO. This is a deliberate design choice: the project prioritizes simplicity and zero-dependency architecture over extensibility.

Full Class Map

All coupling is through constructor injection (composition over inheritance).

Why No Interfaces?

Design Decision: In a production crawler (Heritrix, StormCrawler), you would define interfaces like Frontier, Fetcher, Parser, Storage to enable swapping implementations. This project uses concrete classes because: (1) there is exactly one implementation of each concern, (2) Java interfaces add ceremony but zero value when you have no polymorphism, (3) this is a DAA educational project where algorithm clarity > architectural extensibility.
Concrete Classes (Current)
Direct composition. CrawlEngine takes URLManager, HTTPFetcherHTMLParser, DataStorageModule as constructor args. Simple, readable, zero overhead.
Interface-Based (Alternative)
Define IFrontier, IFetcher, IStorage. Allows mock testing and implementation swapping. Adds 3+ files and indirection. Better for teams > 3 people or if you need Redis-backed frontiers.

Design Analysis

Why maxPages * 10 for frontier size? (Line 29)
The frontier can hold 10× the page budget. With maxPages=100, the frontier caps at 1000. This ratio was empirically determined: Wikipedia pages average ~200 valid links each, so depth 1 from a seed yields ~200 frontier entries. 10× provides comfortable headroom without unbounded growth.

Shutdown Hook (Lines 34-37)

The Runtime.addShutdownHook() ensures that if the user presses Ctrl+C mid-crawl, the DataStorageModule.flush() method still runs, writing CSV/JSON to disk. Without this, all crawl results would be lost.

Shutdown Hook (Current)
Simple, reliable. The JVM guarantees the hook runs on SIGINT/SIGTERM. No extra threading complexity.
Try-Finally + Signal Handler
More explicit but requires sun.misc.Signal (non-standard). Overkill for this project.

URLManager.java 265 lines · BFS Frontier + Deduplication · Critical

Purpose

The brain of the BFS. Manages the URL frontier queue, enforces domain-bounded crawling, performs three-layer deduplication (junk filter → Bloom filter → HashSet), and applies safety caps (maxPages, maxFrontierSize). If you remove this class, the crawler cannot function.

Functions — Detailed Breakdown

public void addSeed(String seedUrl)
Critical — Seeds the BFS frontier with the starting URL. Must be called exactly once before startBFS().
Steps: (1) Normalizes URL via normalizeUrl(). (2) Extracts domain via extractDomain() and saves to seedDomain. (3) Adds to visitedUrls HashSet. (4) Inserts into Bloom filter. (5) Enqueues into frontier.
Edge case: If seedUrl is null, blank, or fails normalization — the method returns silently. The BFS loop will immediately exit because the frontier is empty.
Can it be called multiple times? No. The second call would overwrite seedDomain, breaking domain scoping. A production crawler would accept multiple seeds.
public boolean addIfNotVisited(String url)
Critical — The central gatekeeper. Every extracted link passes through this 7-layer filter chain.
Filter chain (in order, any failure → reject):
① normalizeUrl() — null/blank check + canonical form
② Domain scoping — extractDomain(url) != seedDomain → reject
③ Junk URL filter — isJunkUrl() → reject binaries, namespaces, CMS
④ Page cap — totalEnqueued >= maxPages → reject
⑤ Frontier cap — frontier.size() >= maxFrontierSize → reject
⑥ Bloom filter — mightContain() → fast probabilistic pre-check
⑦ HashSet — visitedUrls.contains() → definitive duplicate check

Bug note: The Bloom filter check on line 104-108 has a subtle logic issue: if mightContain() returns false, the URL is definitely new and proceeds to add. If it returns true, the code checks the HashSet. But if mightContain() returns false, the URL bypasses the HashSet check entirely (correct!). This is the key performance win — Bloom filter short-circuits the expensive HashSet lookup for genuinely new URLs.
private static boolean isJunkUrl(String url)
Important — Content-aware URL classifier. Rejects non-content links that waste crawl budget.
4 filter categories:
1. Binary extensions (regex match): .jpg, .pdf, .css, .js, .mp4, .woff2, etc. (32 extensions). Why regex? A simple endsWith() chain would be 32 lines. The regex is more compact but slightly slower (~2µs per URL). For 10K URLs this is 20ms — negligible.

2. Wikipedia namespaces: Special:, User:, Talk:, Template:, Category:, Help:, Portal:, File:, Module:, MediaWiki:, /w/index.php, action=edit, action=history, oldid=, diff=. These are the #1 cause of BFS explosion on Wikipedia — a single article links to 50+ namespace pages that are not content.

3. CMS junk: /login, /signup, /register, /logout, /wp-admin, /feed, /rss, /print/. These are stateful pages that either require auth or are machine-readable feeds.

4. Query parameter count: URLs with > 2 query parameters are usually dynamic/stateful pages (search results, filter views). Counting & characters is an O(n) string scan.

Can it be removed? No. Without this filter, the crawler wastes 90%+ of its budget on non-content pages. On Wikipedia, this filter alone reduces accepted links from 1830 to 14.
public static String normalizeUrl(String rawUrl)
Critical — Converts any URL to canonical form for deduplication.
4 normalization steps:
① Lowercase scheme and host (HTTP://EXAMPLE.COM → http://example.com)
② Strip default ports (:80 for HTTP, :443 for HTTPS)
③ Remove fragment identifiers (#section) — fragments are client-side only
④ Remove trailing slash (except root /)

Implementation: Uses java.net.URI for parsing, then manually reconstructs. This is more reliable than string manipulation because URI handles edge cases like encoded characters.

Exception handling: Returns null on URISyntaxException. Callers must null-check. This is the correct approach — malformed URLs should be silently dropped by the crawler.

What's missing? (1) Percent-encoding normalization (e.g., %41 → A). (2) Path segment normalization (/a/../b → /b). These would improve dedup accuracy by ~2-3% but add complexity.
private static String extractDomain(String url)
Critical — Extracts the host portion of a URL for domain scoping.
Uses new URI(url).getHost(). Returns null on failure. Why not URL.getHost()? java.net.URL requires a registered protocol handler and is effectively deprecated in modern Java. URI is the correct choice.

public void startBFS(int maxDepth, String keyword)
Critical — The main BFS loop. ~100 lines of the most important code in the project.
Depth tracking algorithm: Uses the "level-order counting trick." Two counters track depth without storing depth per URL:
• pagesInCurrentLevel — counts down as pages are dequeued
• pagesInNextLevel — counts up as child URLs are enqueued
When pagesInCurrentLevel hits 0, advance depth and swap counters.

Loop termination conditions (4):
① Frontier empty → BFS exhausted naturally
② currentDepth > maxDepth → depth limit reached
③ pagesCrawled >= maxPages → page cap hit
④ abortFlag == true → user cancellation

Per-page workflow:
① Dequeue URL ② Sleep for politeness ③ Fetch HTML ④ Extract title + text + links ⑤ Check keyword match ⑥ Create CrawlResult ⑦ Store result ⑧ Enqueue child URLs via addIfNotVisited() ⑨ Print progress line

Why keyword matching is in the engine: CrawlEngine performs the keyword match (lines 92-97) rather than delegating to a separate class because the match requires both the title AND the page text. Moving this to a separate KeywordMatcher class would add indirection for 5 lines of code — not worth it.
public void abort()

Functions

public String fetchPage(String urlString)
Critical — Performs an HTTP GET and returns raw HTML. Returns null on any failure.
7 defense layers:
① URI.create().toURL() — validates URL syntax
② setConnectTimeout() — prevents indefinite TCP wait
③ setReadTimeout() — prevents indefinite body read
④ Status code check (200-399 accepted, 400+ rejected)
⑤ Content-Type check — only text/html or application/xhtml
⑥ Body size cap — reads at most MAX_BODY_SIZE characters
⑦ finally block — connection.disconnect() always runs

Exception handling: Catches IOException (network errors) and IllegalArgumentException (malformed URLs). Both return null. This is the correct behavior for a crawler — failed fetches are silently skipped.

Alternative: java.net.http.HttpClient (Java 11+) — modern, supports HTTP/2, async operations. Not used here because HttpURLConnection is simpler and has been stable since Java 1.1.
public List<String> extractLinks(String html, String baseUrl)
Critical — Regex-based link extractor. Returns all absolute URLs found in the HTML.
Per-link pipeline:
① Regex capture from anchor tag
② Skip empty or "#"-only hrefs
③ Strip fragment (#section) before resolution (crucial!)
④ Skip non-HTTP schemes (javascript:, mailto:, tel:, data:, ftp:, irc:)
⑤ Skip binary file extensions via isBinaryExtension()
⑥ Resolve relative URLs against baseUrl via resolveUrl()

Why strip fragments BEFORE resolving? If the href is "#section" and you resolve it against "http://example.com/page", you get "http://example.com/page#section". Stripping the fragment first gives you nothing (empty href → skip), which is correct — fragment-only links don't point to new pages.
public String extractPageText(String html)
Important — Strips HTML to plain text for keyword matching.
Pipeline: Remove <script> blocks → remove <style> blocks → strip all remaining tags → decode HTML entities → collapse whitespace → cap at 10KB.
The 10KB cap prevents OOM when processing huge pages. Most keyword matching only needs the first few KB of visible text.
private static String decodeEntities(String text)
Moderate — Decodes 9 common HTML entities.
Handles: &amp;, &lt;, &gt;, &quot;, &apos;, &nbsp;, &#39;, &#x27;, &#34;. What's missing? Numeric entities (&#8212; for em-dash, &#169; for ©). A complete solution would use regex to decode &#\\d+; patterns, but these rarely appear in URLs or titles.
private String resolveUrl(String href, String baseUrl)
Critical — Converts relative URLs to absolute using URI.resolve().
Space handling: Replaces spaces with %20 before creating the URI (line 226). Without this, new URI("path with spaces") throws URISyntaxException.
Scheme validation: Only HTTP/HTTPS URLs are returned. FTP, file:, data: are silently dropped.
BloomFilterLayer.java 144 lines · Probabilistic Dedup · Important

Purpose

Memory-efficient probabilistic URL deduplication. Acts as a fast pre-filter before the definitive HashSet lookup. Uses java.util.BitSet and double-hashing (Java's hashCode() + FNV-1a).

Can this entire file be removed? Yes. The crawler would still work correctly using only the HashSet in URLManager. The Bloom filter is a performance optimization — it saves ~850× memory on the dedup fast path. For this educational project with maxPages ≤ 1000, the HashSet alone is fine. For a production crawler handling millions of URLs, the Bloom filter becomes essential.

Fields

FieldTypeMath Behind It
bitArray BitSet Size computed by: m = -(n × ln(p)) / (ln(2))² where n = expected insertions, p = false positive rate. For n=100,000 and p=0.01: m = 958,506 bits ≈ 117 KB.
hashFunctionCount int Computed by: k = (m/n) × ln(2). For the above: k ≈ 7 hash functions. More functions = lower FP rate but slower per-query.

Functions

public boolean mightContain(String element)
Important — Returns false if the element is DEFINITELY not in the filter (safe to add), true if it MIGHT be (check HashSet).
Loops through all k hash functions. If ANY bit position is 0, the element is definitely new. This is the key insight of Bloom filters — false negatives are impossible.
private int getHash(String element, int i)
Critical (internal) — Double-hashing scheme: h_i(x) = (h1(x) + i × h2(x)) mod m.
h1: Java's built-in hashCode() — polynomial rolling hash.
h2: FNV-1a hash — XOR-fold with prime multiplication. Uses magic constants 0x811c9dc5 (offset basis) and 0x01000193 (FNV prime).
Why double-hashing instead of k independent hashes? Kirsch & Mitzenmacher (2006) proved that double-hashing produces the same false positive rate as k independent hash functions. This saves computation: 2 hash computations + k additions instead of k full hash computations.
public double estimatedFalsePositiveRate()
Diagnostic — Runtime estimate of the current FP rate using (1 - e^(-kn/m))^k.
Removable? Yes for production, useful for debugging and tuning. Not called anywhere in the current codebase.

Key Design Decisions

Double-flush guard (line 86): The flushed boolean prevents the shutdown hook and the normal exit path from both calling flush(). Without this, the CSV/JSON files would be written twice — creating duplicates or write contention.
private void flushIncrementalCSV()
Important — Appends only NEW results (since last flush) to a partial CSV file.
Uses lastFlushedIndex to track position. Opens the file in append mode (new FileWriter(filename, true)). The header row is only written on the first flush. This means if the crawler crashes at page 75, you still have pages 1-50 saved.
private static String escapeCSV(String value)
Moderate — RFC 4180 compliant CSV escaping.
Wraps fields in double quotes if they contain commas, quotes, or newlines. Escapes internal quotes by doubling them (" → ""). This is exactly what Excel expects.
private static String escapeJSON(String value)
Moderate — JSON string escaping.
Escapes \, ", \n, \r, \t. What's missing? Unicode escape sequences (\u0000-\u001F) and forward slash escaping. For URL strings, this is sufficient. A production system would use com.google.gson.Gson or org.json.JSONObject.

PriorityScheduler.java 150 lines · Heap-Based Scoring · Moderate

Purpose

Ranks URLs by relevance score using a max-heap (PriorityQueue). Transforms plain BFS (FIFO) into Best-First Search (priority queue). Currently not integrated into the main BFS loop — CrawlEngine uses URLManager's FIFO queue instead. This module is available for future use.

Integration status: This class is complete and tested but NOT called from CrawlEngine. To integrate it, you would replace urlManager.addIfNotVisited(link) in CrawlEngine with scheduler.scoreAndEnqueue(link, anchorText, keyword, depth) and dequeue from the scheduler instead of the URLManager frontier. This is a future enhancement.

Key Functions

public boolean isAllowed(String urlString)
Important — Checks if the crawler is permitted to fetch this URL.
Precedence rules: Allow paths are checked FIRST. If a URL matches an Allow path, it's permitted regardless of Disallow rules. This follows the standard robots.txt specification where Allow overrides Disallow for the same prefix.

Caching: Each domain's robots.txt is fetched and parsed exactly once, then stored in domainRulesCache (HashMap). For a domain-scoped crawler, this means at most 1 robots.txt fetch per crawl session.
private RobotRules fetchAndParse(String domain, String originalUrl)
Important — Fetches robots.txt from a domain and parses it into a RobotRules object.
User-agent matching: Looks for User-agent: * (wildcard) or User-agent: jwebcrawler (specific). Only processes directives within the matching block.
Crawl-delay capping: If a server specifies Crawl-delay: 60 (60 seconds), the code caps it to 30 seconds max to prevent the crawler from waiting forever. If negative, defaults to 1 second.
On failure: Returns RobotRules.empty() — an empty allow-all ruleset. This is the correct behavior per RFC: if robots.txt is unreachable, assume everything is allowed.
public void enforcePoliteness(String urlString)
Important — Sleeps if the last request to this domain was too recent.
Calculates the elapsed time since the last request and sleeps for the remaining gap. Thread interrupt handling: On InterruptedException, re-sets the interrupt flag (Thread.currentThread().interrupt()) per Java best practices.

Integration Note

Currently not called from CrawlEngine. To integrate, add robotsTxt.isAllowed(url) check before fetcher.fetchPage(url) in the BFS loop. Also replace the hardcoded POLITENESS_DELAY_MS with robotsTxt.enforcePoliteness(url).

Design Pattern

This is a utility class (all static methods, no constructor, no state). In Java, this is a common pattern for I/O helpers. The alternative would be a singleton or instance-based approach, but for pure I/O formatting, static is the clearest choice.

SEED_SUGGESTIONS Array (Lines 17-73)

A 14-entry 2D array where each row is {purpose, URL, description}. This is a hardcoded lookup table — not a configuration file. Why not a JSON config? Because reading a JSON file would require either a JSON parser library (dependency) or manual parsing (complexity). A Java string array is compiled, type-checked, and zero-overhead.
