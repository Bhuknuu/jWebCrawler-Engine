1. Project Abstract & Goals

The main goal of this project is to design a functional web crawler that automates the retrieval of web pages to collect data. The project models the internet as a directed graph where web pages act as nodes and hyperlinks serve as directed edges. This structure naturally maps to a Breadth-First Search (BFS) traversal problem, which is one of the core graph algorithms studied in Design and Analysis of Algorithms.

Why This Is a DAA Project: At its core, a web crawler is a graph traversal engine. The internet is a directed graph with ~1.9 billion nodes (web pages) and ~60 billion edges (hyperlinks). Our crawler applies BFS to systematically explore this graph from a seed vertex, using a queue (FIFO frontier) and a visited set (HashSet) — the two canonical data structures of the BFS algorithm.

The crawler accepts a user-supplied seed URL along with a keyword and max pages cap, then systematically explores the web from that starting point using domain-bounded BFS: only URLs on the same host as the seed are followed. At each step it fetches the HTML content of a page, extracts outgoing hyperlinks using regex-based parsing (zero external dependencies), and queues them for future visits. A junk URL filter rejects non-content links (binary files, Wikipedia namespaces, login pages). To ensure the same URL is never processed twice, the project implements a two-layer deduplication system: a fast Bloom Filter pre-check followed by a definitive HashSet lookup.

Results are stored locally as both CSV and JSON files and presented through an interactive CLI interface as a formatted hyperlink table with summary statistics. The system ties together graph traversal, probabilistic data structures, URL normalization, HTTP networking, and interface design into one cohesive pipeline.

2. System Architecture

The architecture follows a pipeline pattern: each module handles one discrete stage. The CrawlEngine (BFS traversal core) orchestrates all other modules, pulling URLs from the frontier, dispatching them to the fetcher, feeding extracted links back through deduplication, and storing results.

2.2 Graph Model of the Internet

Directed Graph G = (V, E) where:
V = Set of all web pages (nodes/vertices). Each page has a unique URL.
E = Set of all hyperlinks (directed edges). An edge (u, v) exists iff page u contains a link to page v.
Seed URL = The source vertex s from which BFS begins.
Depth D = The maximum number of hops from s. BFS at depth D visits all vertices reachable within D edges.

4. Algorithm Deep Dives

4.1 Domain-Bounded BFS

Standard BFS visits ALL reachable nodes — on the web, that's billions. Our engine applies Domain-Bounded BFS: a modified BFS that restricts traversal to the same domain as the seed URL. This is how production crawlers like Googlebot implement focused crawling.

Pseudocode — Domain-Bounded Depth-Limited Web BFS

FUNCTION DomainBoundedBFS(seedUrl, maxDepth, maxPages, keyword):
    frontier ← empty Queue                    // URLManager's LinkedList
    visited ← empty HashSet                   // URLManager's HashSet
    bloom   ← new BloomFilter(maxPages)       // Fast probabilistic pre-check
    domain  ← extractHost(seedUrl)            // Domain boundary
    
    frontier.enqueue( (seedUrl, depth=0) )
    visited.add(seedUrl)
    bloom.insert(seedUrl)
    pagesCrawled ← 0
    
    WHILE frontier is NOT empty:
        IF pagesCrawled ≥ maxPages:             // Hard page cap
            BREAK
        
        (currentUrl, depth) ← frontier.dequeue()
        
        IF depth > maxDepth:
            CONTINUE                           // Depth constraint
        
        page ← HTTPFetch(currentUrl)           // HTTPFetcherHTMLParser
        IF page == null:
            CONTINUE                           // Skip failed fetches
        
        links ← ExtractLinks(page)             // Regex: a[href]
        title ← ExtractTitle(page)
        pagesCrawled++
        
        StoreResult(currentUrl, title, depth)  // DataStorageModule
        
        FOR EACH link IN links:
            IF extractHost(link) ≠ domain:     // DOMAIN BOUNDARY
                SKIP                           // Reject cross-domain
            IF isJunkUrl(link):                // JUNK FILTER
                SKIP                           // Reject binaries, namespaces
            IF bloom.mightContain(link):       // BLOOM PRE-CHECK
                IF link IN visited:
                    SKIP                       // Definitive duplicate
            visited.add(link)
            bloom.insert(link)
            frontier.enqueue( (link, depth+1) )
    
    RETURN results

Complexity Analysis

MetricComplexityExplanation
TimeO(V + E)Each vertex (page) is visited once. Each edge (link) is examined once during extraction.
SpaceO(V)The visited set stores all V vertices. The frontier queue holds at most one level's worth of vertices.
Network I/OO(V)One HTTP request per page. This is the DOMINANT cost — network latency dwarfs all computation.
Why BFS over DFS? DFS would drill deep into one path before exploring siblings. On the web, this means following a chain of links to page 1000 before checking the second link on the seed page. BFS guarantees all pages at depth 1 are found before any at depth 2, giving a more complete picture of the web neighborhood around the seed.

4.2 Bloom Filters

A Bloom Filter is a space-efficient probabilistic data structure that answers set membership queries with a small false positive rate but zero false negatives.

How It Works — Visual

Bit Array Visualization (m = 64 bits, k = 3 hash functions):

= bit set to 1    = bit still 0

Optimal Parameters

Given:
  n = 100,000  (expected unique URLs)
  p = 0.01     (1% false positive rate)

Optimal bit array size:
  m = -(n × ln(p)) / (ln(2))²
  m = -(100000 × ln(0.01)) / (0.6931)²
  m = -(100000 × -4.6052) / 0.4804
  m ≈ 958,506 bits ≈ 117 KB

Optimal hash functions:
  k = (m/n) × ln(2)
  k = (958506/100000) × 0.6931
  k ≈ 6.64 → 7 hash functions
Memory Savings: HashSet storing 100K URLs ≈ 100 MB. Bloom Filter for the same ≈ 117 KB. That's approximately 850x less memory. The trade-off: 1 in 100 new URLs might be incorrectly skipped. For web crawling, this is an acceptable trade-off.

4.3 Heap-Based Priority Queue

The PriorityScheduler transforms plain BFS into Best-First Search by scoring each URL and processing the highest-scored URL next.

Scoring Formula

score(url, anchorText, keyword) =
    5.0 × keywordInAnchor(anchorText, keyword)   // Keyword relevance
  + 2.0 × domainAuthority(url)                    // .edu/.gov bonus
  + 1.0 / (pathDepth(url) + 1)                    // Shorter paths preferred
  + 1.5 × novelDomain(url)                        // New domain bonus

Heap Operations

OperationComplexityPlain BFSPriority BFS
Insert URLO(log n)O(1)O(log n) — heapify up
Get next URLO(log n)O(1)O(log n) — heapify down
Total crawlO(V log V)O(V)O(V log V)

The extra log factor is worthwhile because the crawler processes the most relevant pages within its depth budget instead of blindly following link order.

6. Key Technical Achievements

The most significant engineering decisions that distinguish this crawler from a naive BFS implementation:

Solved Domain-Bounded BFS
Naive BFS follows ALL links, causing exponential explosion on sites like Wikipedia (language links alone create 300+ branches per page). Our crawler extracts the seed's domain and rejects all cross-domain URLs, reducing frontier growth by 95%+. Example: from 1830 extracted links on a Wikipedia page, only 14 passed the domain + junk filter.
Solved Junk URL Filtering
A content-aware URL classifier rejects binary files (.jpg, .pdf, .css), Wikipedia namespace pages (User:, Talk:, Category:, Special:), CMS junk (/login, /wp-admin, /feed), and query-heavy dynamic URLs (>2 parameters). This prevents the crawler from wasting budget on non-content pages.
Solved Two-Layer Deduplication
URL deduplication uses a Bloom Filter as a fast probabilistic pre-check (O(k) time, ~117KB for 100K URLs) before the definitive HashSet lookup (O(1) amortized, ~100MB). The Bloom Filter rejects ~99% of duplicates without touching the heap-allocated HashSet, providing ~850x memory savings on the fast path.
Solved Memory Safety Caps
Three hard limits prevent OOM: maxPages (user-configurable, default 100) caps total pages crawled, maxFrontierSize (10x maxPages) caps the BFS queue length, and MAX_BODY_SIZE (1MB) caps individual page downloads. Together these ensure the JVM never exceeds predictable memory bounds.
Solved URI-Based URL Normalization
All URLs pass through java.net.URI-based normalization: lowercase scheme/host, strip default ports (:80/:443), remove fragments (#section), strip trailing slashes. This prevents duplicate crawling of HTTP://EXAMPLE.COM/page/ and http://example.com/page.
Solved Zero External Dependencies
The entire project uses only the Java standard library (java.net, java.util, java.io). No Jsoup, no Guava, no Maven/Gradle. HTTP fetching uses HttpURLConnection, link extraction uses java.util.regex, Bloom Filter uses java.util.BitSet. Compiles with a single javac command.

8. Challenges & Solutions

Challenge 1: Infinite Crawling Loops
Problem: Without controls, the crawler follows links endlessly. Page A links to B, B links to C, C links back to A — creating an infinite cycle.
Solution: Two-layer protection: (1) The HashSet in URLManager prevents re-visiting any URL. (2) The depth limit in CrawlEngine stops the BFS at a user-defined level, ensuring termination regardless of graph structure.
Algorithm: This is the classic visited array in BFS that converts a potentially cyclic graph into a tree traversal.
Challenge 2: Memory Efficiency at Scale
Problem: Storing millions of visited URLs in a HashSet consumes excessive memory. A Java String averaging 60 bytes × 1M URLs = ~60 MB just for the visited set.
Solution: Bloom Filter integration. A Bloom Filter achieving 1% false positive rate for 1M URLs requires only ~1.2 MB — a 50x reduction. The trade-off (occasionally skipping a new URL) is acceptable for an educational crawler.
Algorithm: Bloom Filters use k independent hash functions mapping to a bit array. The space-accuracy trade-off is controlled by the formula: m = -(n × ln(p)) / (ln(2))².
